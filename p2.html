AI共創型全員経営——教員の声が学校を動かす

P1で起きたこと → だからP2が必要

P1（教員研修）の第2回で、教員はGoogleフォーム→AI統合→KJ法→コンセンサス型評価を体験した。WSでは、全員の意見がAIで構造化され「こんなふうにまとまるのか」という驚きがあった。

そのとき、自然に出てくる問いがある:

「これ、職員会議でもやれないか？」

P2は、その問いに応える。 P1で教員が体験したコンセンサス型意思決定を、学校運営に本格導入するプロセスだ。

世界観との接続

世界観ページで定義した「多様な声が集まるほど意思決定の質が高まる——AI時代の参加型民主主義教育」。P2は、その原理を学校運営レベルで実装するプロセスである。教員が体験したことを、生徒も体験する。3つのレベル（教育委員会・教員・生徒）で同じOSが作動する。

P2で教員が体験するコンピテンシー

P2の4つのフェーズは、AI PROOFコンピテンシーの実践の場でもある:

フェーズ体験するコンピテンシー
フェーズ1: コアチーム立ち上げ1. 仕組み化と再投資 — Vaultで業務を仕組み化し、空いた時間で創造的な仕事に取り組む
フェーズ2: 教員の声を集める4. 知の形式知化と接続 — 教員の暗黙知をAIで構造化し、知恵としてつなげる

実証済みの方法

P2のコンセンサス型意思決定は、長崎県教育委員会（7名、2026年3月）で実証済み。「本当の意味での民主主義が実現していく夢を抱かせる話だった」（馬場さん）。また、企業4社でのAI組織変革（AX）で検証済みの方法論を学校に翻訳している。詳細はAbout Usを参照。

1. なぜ学校組織開発にAIか

1.1 馬場さんの問題意識

「50人の先生がいても取り組むのは2-3人。1日30時間のうち3-4時間しか変わらない。それでは生徒にとって学びが変わったとは言えない。20時間、できれば30時間全部変わってほしい」
——馬場さん（長崎県教育委員会 高校教育課）

この問題は、個々の教員の研修（P1）だけでは解決しない。学校の意思決定の仕組みと教員間の知識の流れを変える必要がある。

1.2 従来の学校組織の構造

核心: 「多くの声が集まるほど、意思決定の質が上がる」ことをAIが実現する。

従来、多くの意見を集めても、統合する手段がなかった。AIが統合を担うことで、多様性が邪魔ではなく資産になる。これが「AI共創型全員経営」の意味。

2. なぜこの方法が動くのか——企業AXとの同型性

この方法は理念ではない。企業4社のAI組織変革（AX）で検証済みの方法論を学校に翻訳したものだ。組織が変わらない構造は、企業も学校も同じだからだ。

3. フェーズ1: 教員コアチームの立ち上げ

3.1 コアチームの選定

条件理由
5-10名多すぎるとセットアップに時間がかかる。少なすぎると事例の多様性が出ない
情報担当教員を含む技術的なトラブルシューティングの拠点
異なる教科から教科横断の視点が生まれる
管理職を1名含む組織開発の意思決定に直結させるため
必ずしも「意欲的」でなくてよい「ふつうの先生」が使えることが、全教員展開の鍵

3.2 環境セットアップ

P1（教員研修）の第3回で、コアチームが自分のVaultを作る。

各教員のPC
├── Obsidian（インストール）
├── Claude Code（インストール）
├── 個人Vault/
│   ├── CLAUDE.md          ← 学校情報・担当教科・方針
│   ├── 授業設計/           ← 授業のアイデア・振り返り
│   ├── 組織開発/           ← 職員会議の記録・レポート
│   └── 研修記録/           ← 事例研究会の気づき
└── Git（版管理）
セットアップの原則

北田が1人ずつセットアップを支援する。 「全員集めて一斉に」ではなく、個別に。教員のPCの状態はバラバラなので、一斉セットアップは失敗する（企業AXでの教訓）。

3.3 コアチームの最初の活動

コアチームが最初にやることは、自分の仕事にVaultを使うこと。

活動内容

事例研究会の記録P1の事例研究会で出た気づきをVaultに記録→次回の事例研究会で前回の記録を参照

核心: コアチームは「モルモット」ではなく「設計者」。 全教員への展開を一緒に考えるメンバーとして位置づける。

4. フェーズ2: AIアプリで教員の声を集める

4.1 サツドラの教訓——ツールの前に「声を聴く」

サツドラ中村CHROの原則

「Obsidian/Claude Codeの導入は『希望を感じてから』——ツールを先に入れない」

「まずポジティブ体験を作る → 自分たちの声が聞かれる取り組みだと感じてもらう」

この原則を学校に翻訳する:

サツドラ（企業）学校
バイヤーは疲弊し「また何かやらされるのでは」とディフェンシブ教員も多忙で「また新しいことを押し付けられる」と警戒
AI壁打ちアプリでバイヤーの悩みを引き出すAIアプリで教員の声を収集する
ログを回収→AIで構造化→レポート化回答を回収→AIで統合→レポート化
「悩み事も集合知です」という文化を示す「先生方の声が学校を変える」という体験を作る

4.2 AIアプリによる情報収集

全教員が使うのはAIアプリ（またはGoogleフォーム）だけ。 この段階ではClaude Code＋Obsidianの導入は不要。コアチームがAIによる統合を裏で担う。

収集の方法

質問の設計

回テーマ質問例狙い
第1回学校の課題「あなたが日々の仕事の中で最も困っていること、変えたいと思っていることを自由に書いてください」教員の「声」を可視化する
第2回生徒について「あなたが担当する生徒の学びについて、気づいていること・心配していること・可能性を感じていることを書いてください」教員一人ひとりが持つ「現場の知恵」を集める
第3回学校の未来「3年後、この学校がどうなっていたら嬉しいですか。理想の姿を自由に書いてください」カリキュラム設計（P3）への布石
質問設計のポイント（サツドラでの学び）
  • 自由記述が鍵。選択式では本音が出ない
  • カバー範囲が広い質問。あまり具体的にしすぎない
  • 5-10分で答えられる量。忙しい教員の負担を最小化
  • 「あなたの声が学校を変える」というメッセージを添える

4.4 レポートの共有と対話

統合レポートを全教員に共有し、3-4人のグループで対話する。

Step内容時間
1. レポート共有統合レポートを配布。「これは皆さんの声をAIが統合したものです」5分
2. 個人で読む自分の意見がどこに反映されているか確認。「おお、こういうふうにまとまるのか」5分
3. グループ対話3-4人で感想を共有。「私もそう思ってた」「こういう見方もあるのか」15分

5. フェーズ3: コンセンサス型意思決定の実装

5.1 フェーズ2からの接続

フェーズ2で「教員の声を集めて統合する」体験をした後、次の問いが自然に生まれる:

「では、この課題にどう取り組むか？」

この問いに対してコンセンサス型意思決定を導入する。

5.2 コンセンサス型のプロセス

3/28長崎研修で実証済みのプロセス

Stepやることツール
1. 叩き台の提示統合レポートから導かれたアクション候補を提示文書配布
2. 評価の入力全教員がQRコード→Googleフォームで1/2/3評価＋理由を入力スマホ/PC
3. AI統合コアチームがClaude Codeで評価を統合。分布と理由を構造化Claude Code
4. AIエージェントの追加「慎重派教員エージェント」「生徒の声エージェント」等の視点を追加Claude Code
5. バージョンアップブロックの声を取り込んだ改訂版を生成Claude Code
6. 再評価改訂版に対して再び1/2/3評価フォーム
7. 記録議論経過と最終合意をVaultに記録Obsidian

5.3 なぜコンセンサス型か——「多くの声＝質の向上」の実感

従来の職員会議コンセンサス型職員会議
声の大きい人の意見が通る全員がフォームで平等に発言
反対意見は「空気を読んで」沈黙ブロック（3）が出ると提案がバージョンアップ→反対意見が改善のエンジン
少数の意見は無視されるAIが全員の意見を統合→少数意見も可視化
「決まったこと」に当事者意識がない自分の意見が反映されたプロセスを経た→当事者意識
議事録は「結論」だけVaultに議論経過が蓄積→「なぜこう決めたか」が後から追える

「ブロックが出るほど提案が良くなる」 という体験が、「多様な意見＝質の向上」の実感を生む。これが組織文化の転換点になる。

5.4 適用する職員会議のテーマ（段階的に拡大）

段階テーマ例難易度
最初の1回事例研究会の進め方低（試しやすい）
2-3回目来年度の行事計画の見直し中
4-6回目探究学習の時間配置中〜高
それ以降カリキュラム改訂、学校評価、組織体制高

5.5 学校運営のAI活用の高度化

コンセンサス型意思決定が定着してくると、学校運営の他の領域でもAI活用が広がる。コアチームがClaude Code＋Obsidianで以下のような業務を高度化し始める。

領域AI活用の内容従来との違い
広報学校の取り組みをClaude Codeで記事化。写真＋テキストからHP・SNS・学校便りを一括生成担当者1人が孤軍奮闘→AIが下書き＋コアチームがレビュー
行事計画過去の行事の振り返りをVaultから参照→改善案をAIが提案→コンセンサス型で決定「去年と同じ」の繰り返し→データに基づく改善
保護者対応よくある質問のパターンをVaultに蓄積→AIが回答案を生成→教員がレビュー毎回ゼロから→蓄積された知恵から回答
教育委員会への報告取り組みの経過をVaultから抽出→レポートをAIが構造化報告書作成に膨大な時間→Vaultに蓄積した記録からAIが生成

この段階で起きること: コアチームの教員が「Claude Code＋Obsidianがあると、自分の仕事がこう変わる」という具体的な成功事例を持っている。これが次のフェーズ4で全教員に環境を展開する際の最強の説得材料になる。

6. フェーズ4: 教員全体への展開と集合知の形成

6.1 展開の原則——「やらされる」ではなく「やりたくなる」

サツドラでの教訓
  • ツールを先に入れない。希望を感じてから
  • 「また何かやらされる」と思われたら終わり
  • 成功事例を見て「自分もやりたい」と思う人から展開する

フェーズ2-3を経て、全教員は以下を体験している:

  • 自分の声がAI統合レポートに反映された
  • コンセンサス型で「自分の意見が学校を動かした」
  • コアチームが広報や行事計画でAIを使いこなしている姿を見ている

この状態で「自分もVaultを持ちたい」という声が出てくる。その声が出てから展開する。

6.2 展開のプロセス

6.3 個人の宇宙×集合知のバランス

サツドラ中村CHROのメタファー

「各自のObsidianで自分の世界を構築→抽出されたエッセンスを共有空間にポイポイ」

「受粉」のメタファー——自分の宇宙に違う花粉が入ってきて展開する

これを学校に翻訳する:

核心: 「混ぜすぎない」こと。 全員が同じVaultを使うのではなく、個人のVaultが独立していて、共有はエッセンスだけ。バイヤーの棚づくりが個人の美的感覚に依存するように、教員の授業設計も個人の創造性に依存する。集合知はその創造性を触発するもの。

6.4 集合知の形成——何が起きるか

段階内容具体例
個人の蓄積各教員がVaultに授業設計・振り返り・気づきを蓄積「この単元は、こういう導入だと生徒の食いつきが違った」
事例の共有事例研究会で実践を発表。AIが統合レポートを生成「先月の事例研究会では、3教科で同じ課題が浮上した」
パターンの発見蓄積されたレポートからAIがパターンを抽出「AI壁打ち型授業は、理科系で効果が高く、国語では意見統合型が合う傾向」
知恵の循環パターンが次の授業設計にフィードバック新任教員がVaultの過去事例を参照して授業を設計できる
文化の形成「実践を共有し、学び合う」ことが学校の文化になる事例研究会が教員主導で自走する
